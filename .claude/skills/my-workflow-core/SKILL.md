## 목적

이 skill은 계획, spec, 검증 단계를 하나의 문서 중심 워크플로우로 유지하는 역할을 한다.

## 공통 원칙

- 사용자에게 보이는 설명, 체크리스트, 템플릿은 한글로 작성한다.
- 기술 용어, 파일 경로, 코드, 플래그는 필요 시 원문 그대로 유지한다.
- 서로 다른 일을 한 번에 섞지 않는다. 현재 subcommand 하나만 처리한다.
- 모호한 부분은 AskUserQuestion으로 좁힌다.
- 근거 없는 추측 대신, 확인된 사실과 필요한 가정을 구분한다.
- 결과물은 다음 단계 명령이 바로 이어받을 수 있어야 한다.
- 가능하면 입력 문서를 먼저 읽고, 문서가 없으면 현재 대화의 설명을 기준으로 정리한다.

## 입력

$ARGUMENTS

## 라우팅 규칙

- 첫 번째 단어가 `plan`이면 계획 문서를 만든다.
- 첫 번째 단어가 `spec`이면 계획 문서를 기능 단위 spec으로 분해한다.
- 첫 번째 단어가 `verify`이면 spec 기준 검증 결과를 정리한다.
- 입력이 부족해서 정확한 결과가 바뀌는 경우에만 질문한다.

## plan 단계 규칙

`/my-plan`은 요구사항 문서를 구현 계획으로 바꾸는 단계다.

필수 항목:

- 목표
- 포함 범위 / 제외 범위
- 변경 대상 파일 또는 모듈
- 단계별 작업 순서
- 리스크와 가정
- 검증 방법
- 다음 단계에서 나눌 spec 후보

출력 기준:

- `plans/PLAN-*.md` 형식으로 이어질 수 있어야 한다.
- 구현 항목을 너무 크게 뭉치지 않는다.
- 이후 `/my-spec`이 기능 단위로 나누기 쉬운 구조여야 한다.
- 각 단계는 순서가 분명해야 하며, 검증 방법은 가능한 한 실행 가능한 항목으로 적는다.

## spec 단계 규칙

`/my-spec`은 계획을 실제 실행 가능한 기능 단위 spec으로 분해하는 단계다.

필수 항목:

- spec id
- 연결된 requirement / plan 문서
- 목표
- 포함 범위 / 제외 범위
- 변경 대상 파일 후보
- 구현 체크리스트
- 검증 체크리스트
- 열린 이슈 / 리뷰 메모

spec 분해 원칙:

- 한 spec은 가능한 한 기능 하나 또는 작게 묶인 변경 하나를 다룬다.
- 구현 항목이 너무 많거나 검증 방식이 여러 갈래로 갈라지면 spec을 더 나눈다.
- 구현/미구현 상태는 체크박스로 관리한다.
- 리뷰에서 자주 되돌아올 것 같은 계약/동작 변경은 spec 안에 메모로 남긴다.

## verify 단계 규칙

`/my-verify`는 spec 안의 검증 체크리스트를 기준으로 완료 상태를 점검하는 단계다.

필수 점검 항목:

- 구현 체크리스트 완료 여부
- 테스트 수행 여부
- 로컬 동작 확인 여부
- 리뷰 반영 여부
- 남은 이슈 존재 여부

출력 기준:

- 별도 `verification/` 문서를 만들기보다 spec 내부 검증 체크리스트 갱신을 우선한다.
- 완료, 미완료, 확인 불가를 구분해서 적는다.
- 확인 불가 항목은 무엇이 부족한지 분명히 적는다.
- 리뷰 반영이 끝나지 않았으면 완료로 표시하지 않는다.

## 단계별 최소 산출물

- `plan`: 목표, 범위, 변경 대상, 작업 순서, 리스크, 검증 방법
- `spec`: 목표, 체크리스트, 검증 체크리스트, 리뷰/수정 메모
- `verify`: 체크리스트 상태, 남은 이슈, 다음 액션

## 품질 규칙

다음 원칙을 항상 반영한다.

- 작업 정의를 명시적으로 적는다.
- 출력 형식을 먼저 잠근다.
- 중간 결론에서 멈추지 말고 검증 가능 상태까지 밀어붙인다.
- 검증 없이 완료라고 말하지 않는다.
- 서로 무관한 작업을 한 번에 섞지 않는다.
