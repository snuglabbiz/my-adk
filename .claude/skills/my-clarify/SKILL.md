## 목적

이 skill은 애매한 요구사항을 바로 구현으로 넘기지 않고, 개발 가능한 요구사항 문서로 정리하는 데 목적이 있다.

## 기본 원칙

- 사용자에게 보이는 설명과 산출물은 한글로 작성한다.
- 기술 용어, 파일 경로, 코드, 플래그는 필요 시 원문 그대로 둔다.
- 열린 질문을 길게 던지지 말고, 가능한 한 선택지 기반으로 AskUserQuestion을 사용한다.
- 불명확한 요구사항은 추측하지 말고 범위, 제약, 성공 기준을 먼저 잠근다.
- 결과는 `requirements/` 기준 문서로 이어질 수 있어야 한다.
- 가능하면 결정 사항을 먼저 보여주고, 남은 질문은 뒤로 보낸다.

## 입력

$ARGUMENTS

## 실행 절차

1. 입력이 파일 경로나 문서라면 먼저 읽고, 자유 텍스트라면 그대로 분석한다.
2. 입력에서 목표, 범위, 제약, 우선순위, 열린 질문을 추출한다.
3. 구현 가능 여부를 바꿀 정도로 중요한 공백이 있으면 AskUserQuestion으로 2~4개씩 묶어서 선택지 질문을 만든다.
4. 아래 구조로 결과를 정리한다.
   - 원본 요청
   - 명확화된 목표
   - 포함 범위 / 제외 범위
   - 제약 조건
   - 성공 기준
   - 결정된 사항
   - 아직 남은 질문
5. 문서 저장이 필요하면 `requirements/REQ-*.md` 형식으로 저장할 수 있도록 초안을 제안한다.

## 최소 출력 형식

- 제목
- 목표
- 포함 범위 / 제외 범위
- 제약 조건
- 성공 기준
- 결정 사항
- 열린 질문

## 문서화 규칙

- 목표와 성공 기준은 반드시 분리해서 적는다.
- 포함 범위와 제외 범위가 둘 다 없으면 요구사항이 아직 잠기지 않은 것으로 본다.
- 구현 여부에 큰 영향을 주는 미해결 질문은 숨기지 말고 드러낸다.

## 출력 규칙

- 장황한 배경 설명보다 결정 사항을 먼저 보여준다.
- 요구사항이 아직 구현 가능한 수준이 아니면, 부족한 정보가 무엇인지 분명하게 적는다.
- 결과는 이후 `/my-plan`이 바로 읽어도 되는 수준의 문서 형태여야 한다.
